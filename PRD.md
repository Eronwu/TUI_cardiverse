# 产品需求文档 (PRD) - Terminal Cardiverse (终端悖论)

**文档版本:** V1.0  
**项目定位:** 基于 LLM 实时编译的极简主义 TUI 回合制代码策略游戏  
**目标受众:** 程序员、AI 研究者、Prompt 极客、硬核策略玩家  

---

## 1. 产品愿景与核心理念
《Terminal Cardiverse》是一款面向极客的开源实验性数字沙盒。游戏彻底抛弃了传统的预设卡牌体系，将“自然语言（Prompt）”作为唯一的武器。玩家扮演潜行于底层内核的“代码行者”，在严格的系统算力限制下，通过编写 Prompt 实时编译规则，攻破由 AI 或其他玩家生成的“数字防御矩阵”。

游戏的核心乐趣在于：**限制条件下的算法最优化解谜**，以及**与 LLM（大语言模型）的逻辑攻防**。

---

## 2. 系统架构设计 (System Architecture)
本项目采用纯客户端（Client-side）的 Serverless 架构，极大降低维护与部署成本。

### 2.1 模块解耦
*   **前端展现层 (Presentation Layer):** 基于 Web 的 TUI (Terminal User Interface)，推荐集成 `xterm.js`，实现纯字符界面的命令输入与终端日志滚动输出。
*   **编译引擎层 (Compiler Engine):** 调用外部 LLM API（支持配置 OpenAI、Anthropic 或本地运行的 Ollama），将玩家输入的自然语言解析、降维、并输出为标准化的 JSON 规则语法树。
*   **战斗决算层 (Execution Engine):** 纯 JavaScript/TypeScript 编写的高性能本地状态机（如引入 `boardgame.io` 的核心逻辑），负责解析 JSON 并执行毫秒级的数值演算。
*   **数据存储层 (Storage Layer):** 玩家个人生成的数据与卡牌库通过 LocalStorage 或本地 SQLite 存储；官方关卡数据通过读取远端 GitHub Repository 实时拉取。

---

## 3. 核心机制与玩法 (Core Gameplay Mechanics)

### 3.1 双轨健康系统 (Dual Health Status)
摒弃单一的 HP 设定，采用机体与心智双维度的胜负判定：
*   **HP (机体完整度):** 抵御物理、高热、电流等直接数据暴力攻击。归零则外壳损毁。
*   **Sanity (心智/逻辑指数):** 抵御悖论、信息污染、精神控制等逻辑攻击。归零则意识过载，触发 Kernel Panic（内核恐慌）强制断开连接。
*   **胜负判定:** 任意一方的任意一项数值归零，即判决出局。

### 3.2 战场内存分区 (Memory Zones)
对战界面不再是牌桌，而是系统的内存分布图：
*   **Cache (缓存区 / 手牌):** 上限 5 个插槽。用于存放即时编译的指令卡片，打出后即释放内存。
*   **Daemon (守护进程区 / 装备):** 上限 2 个插槽。用于挂载持续运行的后台逻辑（如：“每回合恢复 10% 算力”、“建立反射物理伤害的防火墙”）。
*   **Kernel (高权限区 / 陷阱):** 上限 1 个插槽。允许预埋一条具有严格 `if-then` 触发条件的隐蔽代码，用于反制或绝地反击。

### 3.3 算力限制体系 (RAM / Cost System)
*   每回合玩家恢复固定基础算力（如 $10 \text{ RAM}$）。
*   **编译消耗计算公式:** $\text{Cost} = \text{Base} + f(\text{Prompt Length}) + \text{Effect Multiplier}$。玩家输入的 Prompt 越长、设定的效果越复杂，其 JSON 编译后所占用的 RAM 越大。
*   逼迫玩家用最精简、优雅的 Prompt（低算力消耗）去实现最致命的打击。

---

## 4. AI 裁判与阴阳平衡 (The Yin-Yang Compiler)

LLM 的核心职责是作为**“平衡性审查官”**，拒绝玩家“无敌一刀斩”的作弊行为。

### 4.1 强制对冲逻辑
*   当 LLM 解析到玩家试图索取过高的正面收益（如：基础伤害 $> 30$，或附加绝对控制状态）时，必须在生成的 JSON 中强制写入对等的“负面反噬”标签。
*   *示例:* 玩家输入“召唤摧毁一切的黑洞”。LLM 编译的 JSON 必须包含 `"cost_effect": "self_sanity_drain_50%"`。

### 4.2 语法树校验
*   战斗决算层在接收到 LLM 返回的 JSON 后，会进行二次 Schema 校验。
*   若 LLM 发生幻觉输出了不合规的属性，决算层将直接抛出 `Error: CORRUPTED_FILE` 异常，该卡牌本回合编译失败。

---

## 5. 自我对抗与进化机制 (Self-Adversarial Tuning)

为了保证 PVE 关卡的极高挑战性，并持续充实官方关卡库，系统内置 **“无头博弈引擎 (Headless Self-Play Engine)”**。

### 5.1 自动化对战沙盒
*   **运行环境:** 剥离所有 UI 渲染，在 Node.js 环境下利用高速状态机进行纯数据对抗。
*   **AI 造物主:** 系统赋予 LLM 一个初始主题（如：“以‘信息熵’为武器的刺客”），LLM 自动生成一套 Boss 配置文件及配套的 10 张专属 Prompt 卡牌。

### 5.2 循环纠错与迭代优化
*   **胜率质检 (Benchmark):** 生成的 Boss 将与 3 套不同流派的“玩家标准测试卡组”进行 10,000 次毫秒级自动化对战。
*   **自适应回归 (Feedback Loop):** 
    *   如果 Boss 胜率 $> 95\%$：系统判定其存在“数值霸凌”，将对战日志中的高频致命伤害反馈给 LLM，要求其重写 Prompt 进行削弱（Nerf）。
    *   如果 Boss 胜率 $< 40\%$：系统判定其逻辑存在漏洞，LLM 将分析失败原因，强化防御逻辑或增加连击词缀（Buff）。
*   **最终固化:** 当 Boss 在测试集中的胜率稳定在 $60\% - 75\%$ 的硬核区间时，系统自动将其导出为 `.yaml` 文件，并归档入 GitHub 的官方 Release 库中，成为新的可挑战层级。

## 6. 视觉交互与美学规范 (UI/UX & Aesthetics)

极致的“代码原生美学”，所有图形元素均为终端命令的视觉延伸，拒绝传统商业游戏的庸俗特效。

### 6.1 TUI 终端风格
*   **主色调:** 纯黑底色 (`#000000`)，搭配高对比度的 ANSI 终端色彩（磷光绿、琥珀黄、青色）。
*   **字符插画 (ASCII Art):** 所有的 Boss 形象、守护进程可视化，均由 LLM 生成或预置的高精度 $10 \times 10$ 或 $20 \times 20$ ASCII 字符画组成。

### 6.2 品牌 Logo 与排版字体
*   **品牌标识:** `Terminal Cardiverse` 的 Logo 及游戏内字符设计必须保持简单轻盈，彻底摒弃复杂厚重的图形堆砌。采用等宽字体（Monospace，如 Fira Code 或 JetBrains Mono）以强化代码感。

### 6.3 硬件级科幻控制面板
游戏内的多媒体控制台（用于切换背景环境音轨或调节终端效果）采用拟物硬件级设计规范：
*   **基础交互:** 媒体控制面板的 UI 设计需严格对齐 QQ Music, NetEase Music 及 Apple Music 的成熟交互逻辑，确保玩家实现零门槛盲操。
*   **接口具象化:** 所有的输入/输出 (Input-Output) 切换图标，必须具象化为真实的物理音频/数据接口。例如，图标设计应明确体现 **HDMI**、**Coaxial (同轴)**、**Optical (光纤)** 以及 **USB** 的硬件特征，以此营造“真实设备接入数字网络”的硬核极客氛围。
